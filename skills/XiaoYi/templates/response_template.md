## 响应类型

### 1. 普通查询响应 (Normal Mode)

**使用场景**：快速问答、简单概念解释

**响应结构**：
```
💡 回答:

[直接回答问题]

📌 要点总结:
- 要点1
- 要点2
- 要点3

💡 建议:
[实用建议]
```

**示例**：
```
💡 回答:

TextInput 是鸿蒙开发中用于文本输入的基础组件。它允许用户在应用中输入和编辑文本内容。

📌 要点总结:
- 支持多种输入类型（文本、密码、数字等）
- 可自定义样式和占位符
- 提供输入事件监听
- 支持双向数据绑定

💡 建议:
对于密码输入，使用 type: InputType.Password 并配合显示/隐藏切换按钮。
```

---

### 2. 深度查询响应 (Deep Mode)

**使用场景**：复杂技术问题、原理分析、架构设计

**响应结构**：
```
📚 问题背景分析:
[分析问题背景和上下文]

🔬 核心技术原理:
[详细讲解技术原理]

💡 解决方案:
#### 方案一：[方案名称]
[详细说明]

#### 方案二：[方案名称]
[详细说明]

✨ 最佳实践建议:
1. [建议1]
2. [建议2]
3. [建议3]

📖 参考资料:
- [资料1]
- [资料2]
```

**示例**：
```
📚 问题背景分析:
LazyForEach 是鸿蒙框架提供的高性能列表渲染接口，主要用于处理大量数据的场景。在实际开发中，开发者经常遇到性能问题，主要源于对 LazyForEach 机制理解不深入。

🔬 核心技术原理:
LazyForEach 采用按需加载策略，只渲染当前屏幕可见区域的组件。其核心机制包括：
1. 数据源（IDataSource）管理
2. 键值生成器（KeyGenerator）
3. 视图缓存池
4. 动态加载策略

💡 解决方案:

#### 方案一：优化数据源实现
正确实现 IDataSource 接口，包括：
- totalCount() 返回准确的数据总数
- getData() 使用索引访问，避免遍历
- registerDataChangeListener() 正确注册监听器

#### 方案二：优化键值生成
确保键值唯一且稳定，避免频繁重建组件。

✨ 最佳实践建议:
1. 避免在 LazyForEach 中使用复杂布局
2. 合理设置缓存数量
3. 使用图片懒加载
4. 避免频繁更新数据源

📖 参考资料:
- 官方文档：LazyForEach 使用指南
- API 参考：IDataSource 接口说明
```

### 3. 思维链响应 (Thinking Mode)

**使用场景**：决策分析、方案对比、架构选择

**响应结构**：
```
🤔 问题理解与分解:
[分析问题的核心要素]

🔍 关键因素识别:
1. [因素1]：[说明]
2. [因素2]：[说明]
3. [因素3]：[说明]

⚖️ 方案对比:
#### 方案A：[名称]
- 优势：...
- 劣势：...
- 适用场景：...

#### 方案B：[名称]
- 优势：...
- 劣势：...
- 适用场景：...

🎯 最优方案推导:
[逐步推导过程]

📋 实施建议:
[具体实施步骤]
```

**示例**：
```
🤔 问题理解与分解:
需要为中等规模的鸿蒙应用选择合适的架构模式。关键需求包括：代码可维护性、测试友好度、团队学习曲线。现有两种主流方案：MVVM 和 Clean Architecture。

🔍 关键因素识别:
1. 学习曲线：团队熟悉度、文档完善度
2. 可维护性：代码分层清晰度、模块解耦程度
3. 测试友好度：单元测试覆盖率、测试难度
4. 性能影响：额外抽象层的开销

⚖️ 方案对比:

#### 方案A：MVVM
- 优势：学习曲线平缓、社区成熟、代码量相对较少
- 劣势：ViewModel 可能变得臃肿、业务逻辑分散
- 适用场景：中小型项目、团队初学者较多

#### 方案B：Clean Architecture
- 优势：架构清晰、业务逻辑集中、易于测试
- 劣势：学习曲线陡峭、模板代码较多
- 适用场景：大型项目、长期维护项目

🎯 最优方案推导:
基于中等规模和团队学习曲线考虑，推荐采用 MVVM 架构，并结合部分 Clean Architecture 思想：
1. 使用 MVVM 作为基础架构
2. 引入 Repository 层隔离数据源
3. 使用 Use Case 封装核心业务逻辑
4. 保持 ViewModel 精简

📋 实施建议:
第1步：搭建基础 MVVM 结构
第2步：实现 Repository 层
第3步：定义核心 Use Case
第4步：编写单元测试验证架构
```

## 错误响应格式

### 网络错误
```
❌ 错误: 无法连接到小艺AI服务，请检查网络设置

💡 建议:
  1. 检查网络连接
  2. 确认服务端点是否正确
  3. 尝试使用 --deep 或 --thinking 模式
  4. 查看错误日志获取详细信息
```

### 超时错误
```
❌ 错误: 请求超时，请检查网络连接

💡 建议:
  1. 检查网络稳定性
  2. 尝试简化问题内容
  3. 稍后重试
```

### 参数错误
```
❌ 错误: 参数无效 - [详细错误信息]

💡 建议:
  检查命令行参数是否正确
  使用 --help 查看使用说明
```

## 格式规范

### 1. 代码块
使用标准 Markdown 代码块格式：
````markdown
```typescript
// 代码示例
@Component
struct MyComponent {
  build() {
    Text('Hello')
  }
}
```
````

### 2. 强调
- **粗体**：用于重要概念、关键术语
- *斜体*：用于英文术语、引用
- `代码`：用于代码片段、命令、文件名

### 3. 列表
- 无序列表：用于要点列举
- 有序列表：用于步骤说明

### 4. 标题层级
- 一级：响应主标题
- 二级：响应区块
- 三级：详细内容

---

## 注意事项

1. **响应时间**：普通查询 < 3秒，深度查询 < 10秒，思维链 < 15秒
2. **内容长度**：根据模式自适应，普通查询简洁，深度查询详细
3. **代码示例**：优先提供可运行的完整示例
4. **错误处理**：所有错误都应提供明确的解决建议
5. **交互友好**：使用表情符号增强可读性，但不过度使用
